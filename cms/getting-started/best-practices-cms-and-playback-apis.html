介紹

CMS和Playback API均可以訪問您的Video Cloud視頻數據。本主題旨在幫助您了解它們之間的區別以及使用它們的最佳實踐。

CMS和播放API之間的區別

CMS 和回放 API 訪問相同的底層視頻數據。但是，它們之間存在一些關鍵差異，可以確定您在特定情況下使用哪一種。

一般來說，內容管理系統 API用於後端使用，例如將 Video Cloud 與您的 CMS 系統集成。Playback API 旨在供前端使用，以獲取播放器或視頻門戶（Brightcove Player catalog和playlist例如，API 使用 Playback API）。

下表列出了兩個 API 之間的一些主要區別。

CMS vs播放
物品 內容管理系統 API 播放接口
操作種類 創建，讀取，更新，刪除 只讀-無法使用Playback API修改數據
經營範圍 管理視頻數據的各個方面 提取特定的視頻或播放列表，或搜索視頻
驗證 暫時的訪問令牌 永恆的策略密鑰
數據新鮮度 無緩存，始終為最新 緩存長達20分鐘
響應速度 慢點 更快（由於緩存）
使用權 僅限服務器端（禁用 COR） 服務器或客戶端（啟用 COR）
數據 視頻和播放列表請求不包括視頻源 URL；需要第二個請求才能獲得這些 視頻和播放列表請求確實包含視頻源 URL

使用媒體 URL

重要的是要了解，演繹，圖像和其他資產的URL不是固定的。Brightcove會不時重新配置媒體資產的存儲，並且發生這種情況時，特定資產的URL將會更改。如果您依靠頁面或應用程序中這些資產的硬編碼URL，則鏈接有時會斷開。

此外，所有網址都包含一個TTL出於內容安全原因的令牌。這意味著默認情況下 URL 會在 6 小時後過期。令牌的壽命可以延長至 365 天 - 如果您想要更長的令牌，聯繫 Brightcove 支持 .但是請注意，TTL反映了CDN將緩存資產的最長時間，但不能保證URL在令牌過期之前不會更改。

防止媒體鏈接中斷的最佳方法是在運行時使用內容管理系統 API或者播放接口 .

緩存網址

如果不能選擇運行時API請求，那麼我們建議從本地數據緩存中獲取URL，該URL至少每天刷新一次，或者在為您設置的生存時間範圍內刷新TTL令牌，以較短者為準。

參考編號衝突

本部分適用於內容管理系統 API只要。

為了確保引用 ID 的唯一性，內容管理系統 API在對其分配的視頻進行任何操作後，將 ID 鎖定最多 3 分鐘。當您嘗試重試失敗過快的請求時，或者在刪除之前分配給它的視頻後過早嘗試重新使用參考 ID 時，這可能會導致返回 409 錯誤。見錯誤信息參考更多細節。
