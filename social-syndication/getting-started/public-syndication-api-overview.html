介紹

Brightcove Syndication Configuration API 是一個可公開訪問的 API，允許創建、管理和用於從 Video Cloud 帳戶的視頻目錄生成動態源（例如 MRSS 源）的聯合。

還有一個關聯聯合提要 API可用於檢索與聯合相關聯的提要。

可用性

聯合 API 可供所有有權訪問平台 API 的 Video Cloud 客戶使用。

請注意，必須通過本文檔中描述的 API 來創建聯合提要 - 這些操作沒有 UI。

API參考/基本URL /標頭

這配置 API 參考包含有關可用操作、字段和參數的所有詳細信息。

基本網址是：

https://social.api.brightcove.com/v1

所有請求都必須通過Authorization標頭進行身份驗證：

Authorization: Bearer {your_access_token}

有關訪問令牌的信息，請參閱有關身份驗證的下一節。

對於發送請求正文的任何請求，您還必須包含一個Content-Type: application/json標頭。

驗證

要訪問Configuration API，需要指定一個bearer來自的令牌Brightcove OAuth服務在請求的Authorization標頭。各種 API 方法還需要為相關憑證指定以下操作之一（取決於訪問的方法）：

  • video-cloud/social/mrss/read
  • video-cloud/social/mrss/write

這些操作可以通過Studio 管理模塊的 API 身份驗證部分：

如果願意，還可以通過OAuth API創建憑據：

  • 使用cURL創建客戶端憑據
  • 使用郵遞員創建客戶端憑據

聯合資源
場地 類型 只讀 描述
id 細繩 是的 創建聯合組織時生成
name 細繩 沒有 該聯合組織的內部名稱-POST請求所需
content_type_header 細繩 沒有 如果設置，則覆蓋此聯合供稿的供稿服務器返回的Content-Type標頭。否則，提要默認為特定於聯合類型的標頭值
include_all_content 布爾型 沒有 如果為true，則所有目錄視頻都包含在此聯合中
include_filter 細繩 沒有 如果include_all_content為false，則必須指定。價值是一個內容管理系統 API使用搜索字符串CMS API 視頻搜索語法 v2 .所有語法規則都適用-唯一的區別是，搜索字符串作為include_filter價值而不是?query=參數。
請注意，Feed 響應是通過獲取頁面的項目價值 ( 每頁最多 100 個 ) 使用指定的 include_filter 從目錄中提取，然後應用聯合模板。因此，通過過濾項目include_filter比使用通用模板中的邏輯過濾項目更可取，因為它允許返回一整頁的項目，而不是如果模板中的過濾僅匹配獲取的頁面中的少數項目或沒有項目，則可能返回的項目數量可能要少得多從目錄。
type 細繩 沒有 將生成的提要類型。通用類型允許通過上傳的提要模板生成自定義提要。有效值：advanced , google , iphone , ipad , mp4 , itunes , roku , source , universal .POST請求所必需
title 細繩 沒有 此提要的標題。這包含在適用 Feed 類型的 <channel> 標籤內
description 細繩 沒有 此提要的說明。這包含在適用 Feed 類型的 <channel> 標籤內
syndication_url 細繩 是的 此聯合的 MRSS 提要的 URL
destination_url 細繩 沒有 要包含在 <channel> 標記內的適用供稿類型的 URL
keywords 細繩 沒有 以逗號分隔的列表，僅用於iTunes和（可能）通用供稿
author 細繩 沒有 僅用於iTunes和（可能）通用供稿
owner_name 細繩 沒有 僅用於iTunes和（可能）通用供稿
language 細繩 沒有 僅用於iTunes和（可能）通用供稿-小寫的兩個字母的語言代碼，例如"en"
owner_email 細繩 沒有 僅用於iTunes和（可能）通用供稿
category 細繩 沒有 僅用於iTunes和（可能）通用供稿。要指定具有子類別的類別，請用冒號 (:) 分隔它們 - 例如："Business:Business News". "category": "Music"
album_art_url 細繩 沒有 圖片的 URL，僅用於 iTunes 和（可能）通用提要
fetch_sources 布爾型 沒有 對於通用模板，是否獲取視頻源元數據-如果模板不需要此元數據，則可以通過指定以下內容來提高性能false ;如果不存在可用源，則可以為空
fetch_digital_master 布爾型 沒有 對於通用模板，是否獲取視頻數字主數據-如果模板不需要此元數據，則可以通過指定false ;如果不存在數字主機，則可以為空
fetch_dynamic_renditions 布爾型 沒有 對於通用模板，是否獲取視頻動態再現元數據-如果模板不需要此元數據，則可以通過指定以下內容來提高性能false
sort 細繩 沒有 CMS視頻排序說明符，指示所需的提要結果返回順序。CMS支持的值，例如name，reference_id，created_at，published_at，updated_at，schedule.starts_at，schedule.ends_at，state，plays_total， 和plays_trailing_week可以指定。要按降序排序，請在值前加上減號 (-)，即-created_at , 指定, 提要將按最近排序updated_at默認日期。

看CMS API 視頻搜索語法 v2有關詳細信息include_filter財產..所有搜索語法規則均適用-唯一的區別是，輸入的搜索字符串為include_filter價值而不是?query=參數。

錯誤訊息

如果任何API請求失敗，將返回錯誤消息。錯誤響應如下所示：

  [
    {
      "error_code" : "Application error code 1",
      "message" : "Application error message 1"
    }, {
      "error_code" : "Application error code 2",
      "message" : "Application error message 2"
    }
  ]
請注意，如果在 GET 請求中包含請求正文，則請求將失敗並返回錯誤。

建立企業聯合組織

方法：POST

終點：/accounts/{account_id}/mrss/syndications

示例請求正文：

  {
    "name": "my mp4 feed",
    "type": "mp4"
  }

請注意name和type必填字段。其他是可選的。

更新聯合

方法：PATCH

終點：/accounts/{account_id}/mrss/syndications/{syndication_id}

示例請求正文：

  {
    "name": "my new name"
  }

請注意，PATCH請求的請求主體必須不是包括字段（type，id和syndication_url）。

刪除企業聯合

方法：DELETE

終點：/accounts/{account_id}/mrss/syndications/{syndication_id}

獲取帳戶的所有企業聯合組織

方法：GET

終點：/accounts/{account_id}/mrss/syndications

獲得特定的聯合組織

方法：GET

終點：/accounts/{account_id}/mrss/syndications/{syndication_id}

設置通用聯合的模板

方法：PUT

終點：/accounts/{account_id}/mrss/syndications/{syndication_id}/template

示例請求正文：

  <feed header>My Feed Header</feed header>
  {% for asset in assets %}
  <item>
      <title>{{asset.name}}</title>
      <video_info<{{asset.description}}</video_info>
      </item>
  {% endfor %}

上述模板將生成類似於以下內容的提要：

  <feed header>My Feed Header</feed header>
    <item>
      <title>Title for Video 1</title>
      <video_info>Description for Video 1</video_info>
    </item>
    <item>
      <title>Title for Video 2</title>
      <video_info>Description for Video 2</video_info>
    </item>

獲取通用聯合組織的模板

方法：GET

終點：/accounts/{account_id}/mrss/syndications/{syndication_id}/template

液體的延伸

toUTC過濾器

我們擴展了 Liquid 解析器以支持 toUTC 過濾器，該過濾器將解析大多數標準 ISO-8601 日期時間字符串格式並將它們轉換為標準 UTC 日期時間字符串。Liquid 的日期過濾器可以接受這種格式，然後可以使用它以任何所需格式將時間戳重新格式化為日期時間字符串。例如：

  <pubDate>{{asset.published_at | toUTC | date: "%a, %d %b %Y %H:%M:%S %Z"}}</pubDate>

如果 asset.published_at 的值為 2019-08-09T13:32:52.031Z，則會產生如下輸出：

  <pubDate>Fri, 09 Aug 2019 09:32:52 +0000</pubDate>

toEpoch過濾器

我們使用的 Liquid 解析器不支持日期過濾器中用於將日期轉換為 Unix 紀元時間戳的“%s”標記。為了解決這個問題，提供了一個 toEpoch 自定義過濾器，可用於將有效的日期規範轉換為紀元格式。過濾器返回一個數字數據值，表示自紀元以來的毫秒數，適用於數學過濾器的輸入。例如：

  <toEpochMillis>{{"now" | toEpoch }}</toEpochMillis>
  <toEpochSeconds>{{"now" | toEpoch | divided_by : 1000 }}</toEpochSeconds>
  <thirtyDaysAgo>{{'now' | toEpoch | minus:2592000000 | fromEpoch }}</thirtyDaysAgo>

產生如下輸出：

  <toEpochMillis>1580917253024</toEpochMillis>
  <toEpochSeconds>1580917253</toEpochSeconds>
  <thirtyDaysAgo>2020-01-06T15:40:53.055Z</thirtyDaysAgo>

fromEpoch過濾器

fromEpoch 過濾器將表示自紀元以來的毫秒數的數字轉換為 UTC 日期字符串。過濾器還將接受包含時代值數字的字符串作為輸入。如有必要，可以將輸出傳遞給日期過濾器進行重新格式化。

例如：

  
  <fromEpochMillis>{{"now" | toEpoch | fromEpoch }}</fromEpochMillis>
  <thirtyDaysAgoAltFormat>{{1580917253024 | fromEpoch | date:"%Y-%m-%d" }}</thirtyDaysAgo>

產生如下輸出：

  
  <fromEpochMillis>2020-02-05T16:09:37.809Z</fromEpochMillis>
  <thirtyDaysAgoAltFormat>2020-02-05</thirtyDaysAgo>
