使用 Git：概述

Git是交付系統的重要組成部分，我們鼓勵所有尚未熟悉其概念的用戶了解更多信息。 Git網站。還有一些好用的Git備忘單在那裡使用。

必須安裝Git才能使用Delivery System API，您可以在以下位置進行操作Git網站。我們給出的命令行示例通常不能直接使用，但是，因為使用了許多佔位符。在環境變量或值中替換以下內容： 帳戶ID， REPO_NAME， ACCESS_TOKEN， 用戶名

使用 Git：授權

與 REST API 不同，您只能通過 Git 授權一種方式：使用 Brightcove 用戶名和密碼通過基本身份驗證。當前無法在Git中使用OAuth訪問令牌。

Git應該提示您輸入用戶名和密碼，因此在下面的示例中您將看不到任何身份驗證內容。

Git：創建/更新倉庫

安裝 Git 並通過 REST 創建存儲庫後，您可以開始創建本地存儲庫以供使用。只需創建一個目錄，然後使用git init命令。

          mkdir my_repo
          cd my_repo
          git init

初始化存儲庫後，您現在需要將它綁定到 Brightcove 系統中的遠程存儲庫，以便您可以輕鬆地將更改推送到服務器。

          git remote add origin https://repos.api.brightcove.com/v1/accounts/[ACCOUNT_ID]/repos/[REPO_NAME]

然後，您可以隨意添加、編輯或刪除本地遠程文件。然後，您可以使用普通的 Git 命令來更新遠程存儲庫。

          git add -A
          git commit 'Changing stuff'
          git push

使用 Git：推回購

推送不同文件的位置的詳細信息在輸出中給出推 .您還可以在對存儲庫 URL 的 REST API 調用中找到任何存儲庫的基本 URL。所有文件都存儲在CDN上，以便所有用戶可以快速查看它們。

如果要構建自己的完全自定義的播放器，則應注意，我們不能保證對所有文件的更新將在典型客戶端（例如瀏覽器）中同時進行。因此，強烈建議交付系統 API 的用戶使用版本控制策略，以確保在更新後一起請求緊密耦合的文件。實現此目的的一種策略是在全新的位置進行更新，而不是覆蓋現有文件。這強制要求請求的文件是原始來源，因為我們的服務中不可能有緩存的副本。但是，您應該預計對這些未緩存副本的前幾個請求需要比平時更長的時間。需要明確的是，如果您使用播放器管理 API 而不是直接使用傳送系統 API 創建播放器，則無需擔心這一點，因為緩存問題已為您處理。

您看到實時網站上反映的更新所需的總時間取決於多種因素。最重要的是，這些因素包括瀏覽器緩存以及從我們的邊緣節點完成清除請求所需的時間。通常，從上次推送到存儲庫的時間不應超過五分鐘。這是因為我們提供的文件默認設置為在瀏覽器中緩存五分鐘，而我們所有的邊緣節點平均只需大約一分鐘即可清除。但是，在高峰服務負載下，吹掃時間可能會更長-多達10分鐘。在最壞的情況下，所有緩存層都將需要清除（仍然確定這次），您的更新才能最終生效。除非清除請求失敗或超時並且需要我們的回退緩存處理程序來刷新您的內容，否則這種情況永遠不會發生。

使用 Git：克隆回購

您可以將 Brightcove 已存儲的存儲庫複製到本地系統。用Git術語來說，這稱為克隆存儲庫。這使您不僅可以獲取組織中其他人創建的存儲庫，還可以檢索播放器管理 API 創建的存儲庫。

          git clone https://repos.api.brightcove.com/v1/accounts/{account_id}/repos/$REPO_NAME/$ACCESS_TOKEN

錯誤回應：吉特

Git 調用的錯誤響應僅限於您的 Git 客戶端可以支持的任何內容：

  • 如果您嘗試調用一個不存在的倉庫，通常會收到一條有關以下內容的消息：找不到git-upload-pack：您是否在服務器上運行了git update-server-info？。檢查以確保回購URL正確。
  • 如果您的電話無法通過驗證或授權，通常會提示您輸入密碼。退出此密碼請求，然後嘗試再次瀏覽OAuth指南，以確保您具有有效的訪問令牌。
