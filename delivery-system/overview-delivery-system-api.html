概述

與交付系統關聯的文件通過REST API以及（可選）通過Git進行管理。這些文件被傳送到player.brightcove.net通過 API 調用或“git push”。

大多數 Brightcove 播放器的創建和發布都不需要 Delivery System API。此 API 僅建議高級開發人員使用。

如果您還沒有經歷過一步步：投放系統API，強烈建議您從此處開始。您將進行安全設置，了解系統的一些基礎知識，然後準備好進一步深入了解。

另請參閱API參考。

交付系統REST API

交付系統 API 以存儲庫為中心，也稱為存儲庫。您可以通過一組 REST API 來管理存儲庫，這些 API 允許您添加、獲取和列出您的存儲庫。您還可以使用 REST API 調用來操作存儲庫中的文件。

您可以使用命令行工具 curl 來使用 REST API。REST API 以 JSON 格式返迴響應，其中包含以下有關成功的信息：

  • 名稱：回購名稱。這與在 REST API URL 中找到的名稱相同。
  • public_url：可以看到 repo 文件的基本 URL。

這一步步：輸送系統提供了使用交付系統API的動手指南。

驗證

您可以使用任一方式授權自己OAuth 訪問令牌或使用您的 Brightcove 用戶名和密碼通過基本身份驗證。標準 OAuth 訪問令牌應用於 API 的任何編程使用，但基本身份驗證路由對於命令行使用和入門要容易得多。

我們將在一步步例子。如果您想使用訪問令牌，請更改：

    --user $EMAIL

到：

    --header "Authorization: Bearer $ACCESS_TOKEN"

API錯誤響應

當 REST API 調用出現問題時，有許多可能的錯誤響應以 JSON 格式返回：

  • 如果您嘗試調用一個不存在的API（包括拼寫現有的API調用），則會收到狀態碼404和有關您可能做錯了什麼的有用消息。
  • 如果您的呼叫無法通過身份驗證或授權，您將獲得403狀態代碼。再次嘗試查看 OAuth 指南以確保您擁有有效的訪問令牌。
  • 如果您嘗試使用GET或PUT以外的方法，則響應將包含405狀態代碼。
  • 如果處理您的請求有問題，您將獲得500狀態代碼和標準的Brightcove錯誤響應。

上面提到的標準 Brightcove 錯誤響應是一個包含 error_code 和消息屬性的 JSON 響應。錯誤代碼將是以下之一：

  • INVALID_LOGIN_CREDENTIALS：使用基本身份驗證時提供的用戶名和密碼無效
  • ACCESS_TOKEN_ERROR：給出的訪問令牌無效
  • INVALID_API_CALL：API調用的格式不正確
  • NOT_FOUND：API調用未指向已知資源或未正確格式化
  • NO_PUT_CONTENTS：必須使用包含內容密鑰的多部分錶單來放置回購文件
  • CREATE_REPO_ERROR：創建存儲庫時發生錯誤
  • GET_REPO_ERROR：獲取回購時發生錯誤
  • GET_REPOS_ERROR：獲取所有存儲庫時發生錯誤
  • UNCAUGHT_ERROR：未捕獲的錯誤導致事情失敗
  • UNKNOWN_ERROR：發生了沒有相關錯誤代碼的錯誤

局限性

存儲庫不適合或設計用於非常大的文件。換句話說，請不要在其中存儲視頻。
